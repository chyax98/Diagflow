## 核心语法
- 声明:
  * 语法: mindmap
  * 紧跟根节点

- 根节点:
  * 每个思维导图只有一个根节点
  * 使用形状包裹
  * 示例: root((项目规划))

- 层级缩进:
  * 每层级 +2 个空格（禁用 Tab）
  * 建议最大层级 ≤4 层
  * 同级节点空格数必须完全一致

- 节点形状:
  * root((文本)) - 圆形（通常用于根节点）
  * id(文本) - 圆角矩形
  * id[文本] - 方形矩形
  * id)文本( - 云形
  * id))文本(( - 爆炸形（Bang）
  * id{{文本}} - 六边形
  * 不加形状 - 默认矩形
  * 注意: 形状定义需要在节点文本前加节点ID

- 节点文本:
  * 长度 ≤20 个中文字符
  * 使用关键词，不用完整句子
  * 简洁明了

- 图标（可选）:
  * 语法: ::icon(fa fa-tasks)
  * 使用 Font Awesome 或 Material Design 图标
  * 放在节点文本后
  * 示例: 任务管理::icon(fa fa-tasks)

- CSS 类（可选）:
  * 语法: :::类名
  * 多个类用空格分隔: :::urgent large
  * 需要站点管理员提供 CSS 定义

## 层级结构规则
- 根节点: 1 个
- 一级子节点: 3-7 个（最佳）
- 二级子节点: 每个一级下 2-5 个
- 避免单子节点（孤立分支）

- 缩进一致性:
  * 同级节点空格数必须完全一致
  * 缩进不清晰时，Mermaid 会自动修正到最近的有效父节点
  * ❌ 缩进混乱会导致层级关系不清晰
  * ✓ 使用编辑器的空格可视化功能

- Markdown 格式支持:
  * 使用反引号包裹: \`**粗体** *斜体*\`
  * 自动换行: 文本过长会自动换行
  * 手动换行: 使用换行符

## 分支平衡建议
- 避免某分支过大（>其他分支总和）
- 避免某分支过小（只 1 个子节点）
- 同级节点数 ≤7 个（符合认知规律）
- 深度和广度平衡

## 设计原则
- 中心主题明确
- 分支逻辑清晰
- 关键词精炼
- 层级适中（3-4 层）
- 视觉平衡

## 典型应用场景
- 知识体系: 学科知识、技术栈
- 项目规划: 任务分解、里程碑
- 头脑风暴: 创意收集、方案对比
- 会议笔记: 要点梳理、行动项
- 学习笔记: 章节总结、概念关系

## Kroki 限制
- ✓ 支持所有节点形状
- ❌ 不支持图标语法：`::icon(fa fa-xxx)` 会报 "Parse error"
- ⚠️ 总节点建议 ≤50 个
- ⚠️ 层级建议 ≤5 层
- ⚠️ 节点文本 ≤30 字符

常见错误排查：
1. 注释语法错误
   ❌ // 这是注释（错误：Mermaid 不支持 //）
   ❌ # 这是注释（错误：Mermaid 不支持 #）
   ✓ %% 这是注释

2. 缩进不一致
   ❌ mindmap
        root((中心))
          节点A
           节点B
   ✓ mindmap
        root((中心))
          节点A
          节点B

3. 使用 Tab 缩进
   ❌ 使用 Tab 键缩进
   ✓ 使用 2 个空格缩进

4. 根节点多个
   ❌ mindmap
        root1((中心1))
        root2((中心2))
   ✓ mindmap
        root((中心))
          分支1
          分支2

5. 节点形状不匹配
   ❌ (文本))（左右括号不匹配）
   ✓ (文本)

6. 层级过深
   ❌ 6-7 层嵌套
   ✓ 3-4 层嵌套


## 示例

### 示例 1

```mermaid
mindmap
  root((项目规划))
    需求分析
      用户调研
      竞品分析
      功能清单
    系统设计
      架构设计
      数据库设计
      接口设计
    开发实施
      前端开发
      后端开发
      测试验证
```
